Цель — приблизить отзывчивость к AnyDesk/Parsec, оставаясь на Python.
Реалистичная планка: локально/по хорошему интернету **задержка 30–80 мс** и
плавная картинка без «квадратов».

## Что сейчас тормозит (привязка к коду)

| Узкое место | Где | Чем плохо |
|---|---|---|
| CPU-захват экрана | `host.py` `ScreenStreamer.capture` (`mss`) | BitBlt + хэш кадра в Python на каждый цикл |
| Внутрикадровый JPEG по плиткам | `host.py` `dirty_tiles` (PIL, 4:4:4) | нет межкадрового сжатия → ×5–15 трафика, «квадраты», высокий CPU |
| Детект изменений в Python | `host.py` хэши плиток/кадра | сериализует цикл, упирается в GIL |
| Транспорт TCP | `common.py`, `relay.py` | head-of-line blocking: одна потеря пакета тормозит весь поток |
| Декод по плиткам + smoothscale каждый кадр | `client.py` `apply_tiles`, `run_ui` | PIL-декод на плитку + softscale на CPU |

Вывод: основной выигрыш даёт замена **захвата** и **кодека**; транспорт —
вторым шагом.

## Фаза A (Уровень 1): GPU-захват + аппаратный H.264 поверх текущего TCP/relay

Меняем только горячий путь «захват → кодирование → декод». Шифрование,
relay, ввод, буфер обмена, передача файлов — **без изменений**.

### A0. Зависимости
- `requirements.txt`: добавить `dxcam` (DXGI Desktop Duplication, Windows) и
  `av` (PyAV — биндинг FFmpeg). `mss` оставить как фолбэк.

### A1. Захват — `host.py`
- Новый класс `ScreenCapture`: `dxcam.create(output_idx, output_color="RGB")`,
  `grab()` отдаёт numpy-кадр **только если он изменился** (dxcam сам так умеет —
  убираем ручной хэш кадра и плиток).
- Фолбэк на `mss`, если `dxcam` недоступен (не-Windows / нет GPU).
- Сохранить логику `set_monitor`, `scale`, маппинг ввода (`real_w/real_h`).

### A2. Кодирование — `host.py`
- Новый класс `VideoEncoder` поверх PyAV с авто-выбором энкодера:
  `h264_nvenc` → `h264_qsv` → `h264_amf` → `libx264` (фолбэк, софт).
- Низколатентные настройки: `bf=0` (без B-кадров), `g`=интервал ключевых
  кадров (напр. 120), `tune=zerolatency`/`ull`, целевой **битрейт** вместо
  «качества». GUI-«чёткость» маппим в битрейт (напр. 2–10 Мбит/с).
- На вход — numpy RGB → `VideoFrame` (`yuv420p`/`nv12`), на выход — пакеты.
- Принудительный keyframe при подключении клиента и по запросу (на потерю/реконнект).

### A3. Протокол — `common.py`
- Новые типы: `MSG_VIDEO_INFO` (codec, w, h, fps) и `MSG_VIDEO` (байты пакета
  + флаг keyframe). Старый `MSG_TILES` оставляем для фолбэка.
- В `HELLO` клиент сообщает, умеет ли декодировать видео; host выбирает
  видео-путь, иначе откатывается на плитки. Так старые сборки не ломаются.

### A4. Конвейер — `host.py` `serve`
- Три стадии в отдельных потоках с очередями `maxsize=1` (стратегия
  «последний кадр побеждает», старые дропаем): **захват → кодирование → отправка**.
- Убирает текущий `sleep(frame_interval)` как единственный регулятор — fps
  держим, но не блокируем конвейер.

### A5. Декод и рендер — `client.py`
- Новый `VideoDecoder` (PyAV): пакеты → `VideoFrame` → RGB ndarray → один
  pygame-surface. Заменяет `apply_tiles`.
- Рендер: не звать `smoothscale` каждый кадр без нужды; масштабировать только
  когда размер окна ≠ размеру кадра; рассмотреть отображение в нативном
  разрешении. Статистика FPS/RTT/битрейт — оставить.

### A6. GUI — `app.py`
- Задействовать уже существующее поле `codec` (combobox: Авто/NVENC/QSV/Софт).
- «Качество» → пресет битрейта; FPS/масштаб оставить.

### A7. Тесты
- Обновить `smoke_test.py`: прогон encode→decode без сети, замер времени
  кодирования/декода и размера пакета vs текущий JPEG-путь.

**Итог фазы A:** ожидаемо 20→40–60 fps, кратное падение трафика, исчезают
«квадраты». Уже «комфортно для работы» по LAN/хорошему каналу. Риск низкий —
всё откатывается на старый путь.

## Фаза B (Уровень 2): транспорт на WebRTC (aiortc)

UDP + контроль перегрузки + NAT-traversal одним куском. `relay.py`
превращается из байт-пайпа в **сигнальный сервер**.

### B0. Зависимости
- `aiortc` (+ `aioice`). Для симметричного NAT — внешний TURN (coturn) на VPS
  (ops-задача, не код).

### B1. Сигналинг — `relay.py` → `signaling.py`
- Вместо слепого пайпа: обмен SDP offer/answer и ICE-кандидатами по ID комнаты
  (строчный протокол, как сейчас регистрация). Пароль гейтит комнату.

### B2. Host
- `RTCPeerConnection` + `VideoStreamTrack`, отдающий кадры из `ScreenCapture`.
- Ввод/буфер/файлы — на `DataChannel` (надёжный упорядоченный для файлов/буфера;
  для мыши/клавиатуры — тоже упорядоченный). Заменяет AES-GCM-кадрирование из
  `common.py` на этом пути (WebRTC шифрует сам — DTLS-SRTP).

### B3. Client
- `RTCPeerConnection`, приём видео-трека → тот же рендер, что в A5; DataChannel
  для ввода вместо `FrameSender`.

### ⚠️ Ключевое решение фазы B (аппаратный кодек vs простота)
`aiortc` кодирует H.264 **своим софтовым кодеком** (openh264/x264) — NVENC из
коробки не использует. Варианты:
- **B-простой:** отдать кодирование aiortc (софт). Получаем UDP + congestion
  control + NAT, но теряем аппаратный энкодер из фазы A. Проще, чуть выше CPU.
- **B-полный:** скармливать aiortc уже закодированные NVENC-пакеты через
  кастомный track. Сохраняет hardware-encode + транспорт WebRTC, но заметно
  сложнее.

Рекомендую начать с **B-простого** на 1080p (часто хватает), и только если
упрёмся в CPU/качество — делать B-полный.

**Итог фазы B:** устойчивость к потерям и джиттеру, работа через NAT без
проброса портов, целевые 30–80 мс по интернету.

## Порядок и оценка

1. **Фаза A** — наибольший выигрыш на усилие, низкий риск, всё на месте.
   ~1–2 дня. Делаем и **замеряем** (fps, мс на кодек, КБ/кадр vs сейчас).
2. **Фаза B** — после подтверждённого выигрыша A. ~несколько дней + настройка
   TURN. Сначала B-простой.

## Чем мерить успех
- FPS и RTT (уже есть оверлей в `client.py`).
- Время кодирования/декода на кадр (добавить в `smoke_test.py`).
- Битрейт КБ/с при движении и в покое.
- Субъективная задержка «мышь → отклик».
